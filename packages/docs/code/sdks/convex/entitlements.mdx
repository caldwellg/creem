This page covers reading billing state — gating features on it, evaluating usage limits, and the account-level widgets. Setup is in the [Quickstart](/code/sdks/convex/quickstart).

## The billing snapshot

`snapshot` (generated by `creem.api({ resolve })`) and `creem.getBillingSnapshot(ctx, { entityId })` return the canonical billing state:

```ts
{
  entityId: "org_123",
  subscriptions: [/* Creem recurring subscriptions, incl. add-ons */],
  orders: [/* one-time orders */],
  appPlanAssignments: [/* app-owned plans: free, no-card trials */],
  access: [/* derived: everything the entity currently has */],
  paymentRecoveryState: "none",
  availableBillingActions: ["portal", "cancel"],
}
```

The mental model:

- `subscriptions` mirrors Creem recurring subscriptions and supports multiple simultaneous rows (base plan + add-ons).
- `orders` mirrors Creem one-time orders.
- `appPlanAssignments` stores component-owned current or scheduled app plans (free plans, no-card trials).
- `access` is a **derived read model** combining active subscriptions, paid one-time orders, and active app-plan assignments — convenient for entitlement checks, but not a source of truth.

Because everything is synced into Convex, reading the snapshot is a local, reactive query — no Creem API round-trip.

## Gating UI with BillingGate

`BillingGate` conditionally renders based on which billing actions are available in the snapshot (`checkout`, `portal`, `cancel`, `reactivate`, `switch_interval`, `update_units`, `contact_sales`):

<Tabs>
  <Tab title="React">

```tsx
<BillingGate
  snapshot={snapshot}
  requiredActions="portal"
  fallback={<p>Upgrade to access the billing portal.</p>}
>
  <p>You have portal access.</p>
</BillingGate>
```

  </Tab>
  <Tab title="Svelte">

```svelte
<BillingGate snapshot={snapshot} requiredActions="portal">
  {#snippet children()}
    <p>You have portal access.</p>
  {/snippet}
  {#snippet fallback()}
    <p>Upgrade to access the billing portal.</p>
  {/snippet}
</BillingGate>
```

  </Tab>
</Tabs>

## Usage limits

Catalog plans can carry `limits` (arbitrary numeric keys like `projects` or `aiMessages`). `evaluateUsageLimits` compares your app's usage counters against the active plan's limits:

```ts
import { evaluateUsageLimits } from "@creem_io/convex/react"; // or /svelte

const usageLimits = evaluateUsageLimits({
  catalog: billingCatalog,
  planId: activePlanId,
  usage: {
    projects: 3,
    aiMessages: 72,
  },
});
```

The library evaluates limits; **your app owns the actual usage counters** and where they're stored.

<Warning>
  UI gates are user experience, not security. Protected backend actions must check billing state
  server-side — read the snapshot inside your Convex function before doing the work.
</Warning>

## UI permissions

`BillingPermissions` controls which widget buttons are enabled — useful for role-based UI like "only org admins manage billing". Set once at the provider:

```tsx
<CreemConvexProvider
  api={billingApi}
  permissions={{
    canCheckout: isAdmin,
    canChangeSubscription: isAdmin,
    canCancelSubscription: isAdmin,
    canResumeSubscription: isAdmin,
    canUpdateUnits: isAdmin,
    canAccessPortal: isAdmin,
  }}
>
  ...
</CreemConvexProvider>
```

Disabled permissions render greyed-out buttons. This is cosmetic gating — enforce the real rules in your Convex functions ([Advanced → Custom auth and RBAC](/code/sdks/convex/advanced#custom-auth-and-rbac)).

## Account widgets

### Billing portal

Opens the Creem customer portal (payment methods, invoices). Auto-hides when the entity has no Creem customer record yet — customers are created on first checkout:

```tsx
<BillingPortal />
<BillingPortal>Manage billing & invoices</BillingPortal>
```

Requires `customers.portalUrl` in the connected API.

### Billing history

Paginated transaction history backed by Creem's transaction search (transaction rows only — not invoice documents):

```tsx
<BillingHistory pageSize={5} />
```

Requires `transactions.search` in the connected API:

```ts
const billingApi: ConnectedBillingApi = {
  uiModel: api.billing.uiModel,
  transactions: { search: api.billing.transactionsSearch },
  // ...
};
```

### Payment recovery

When a subscription is past due or a payment failed, surface it and route the user to the portal to fix their payment method:

```tsx
<PaymentRecoveryBanner snapshot={snapshot} />
<PaymentRecoveryButton portalUrl={billingApi.customers!.portalUrl!} />
```

The banner derives its state from `snapshot.paymentRecoveryState` (or accepts an explicit state); the button needs `customers.portalUrl`.

### Checkout success

Show a confirmation banner when the user returns from checkout — Creem's query parameters are parsed automatically:

```tsx
<CheckoutSuccessSummary />
```

React additionally exports a `useCheckoutSuccessParams()` hook returning the parsed params.

### Status banners

Presentational components for specific states, fed from the snapshot:

| Component               | Shows                                                                 |
| ----------------------- | --------------------------------------------------------------------- |
| `ScheduledChangeBanner` | Scheduled cancellation or pending period-end update, with undo/resume |
| `TrialLimitBanner`      | Trial expiration notice                                               |
| `PaymentWarningBanner`  | Pending, refunded, or partially refunded payment warning              |

## Server-side entitlement checks

The same snapshot powers backend enforcement. A typical pattern:

```ts
// convex/projects.ts
export const createProject = mutation({
  args: { name: v.string() },
  handler: async (ctx, args) => {
    const user = await requireUser(ctx);
    const snapshot = await creem.getBillingSnapshot(ctx, {
      entityId: user.orgId ?? user._id,
    });
    const hasPaidAccess = snapshot.access.some(
      (a) => a.kind === "subscription" && a.status === "active",
    );
    const projectCount = await countProjects(ctx, user.orgId ?? user._id);
    if (!hasPaidAccess && projectCount >= FREE_PROJECT_LIMIT) {
      throw new ConvexError("Upgrade to create more projects");
    }
    // ...create the project
  },
});
```

Widget gates keep the UI honest; this is what actually protects the feature.
